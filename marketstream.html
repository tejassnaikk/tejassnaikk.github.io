01

The problem

High-frequency order-book data arrives fast, bursty, and noisy. Building a model on it is the easy part; the hard part is everything around the model — ingesting a live feed without dropping messages, computing features in real time, serving predictions, and knowing when the model silently starts to drift. Most student projects stop at a notebook with a good offline score. The goal here was the opposite: a system that actually runs continuously and tells you when it's unhealthy.

02

Architecture

The data path is a straight line; the interesting parts are the cross-cutting concerns that make it production-grade. Everything below runs on one EC2 instance.

Data path

Binance WS Kafka DuckDB · features LightGBM FastAPI Streamlit
Observability — Prometheus scrapes metrics, Grafana dashboards
Alerting — drift detection → AWS SNS notifications
Tracking — MLflow (SQLite backend) for model runs
CI/CD — GitHub Actions for build, test, retrain

▸ all of the above hosted on a single AWS EC2 instance (t3.small) + S3

Kafka decouples ingestion from processing: the WebSocket consumer can keep up with bursts while downstream feature computation reads at its own pace. DuckDB computes streaming features in-process. LightGBM scores them, FastAPI serves the prediction, and Streamlit visualizes it. Prometheus/Grafana watch the system, a drift check fires SNS alerts when inputs shift, and GitHub Actions automates the retrain-and-deploy loop.

03

Key decisions & tradeoffs

These are the choices that show engineering judgment — each one is a place where the obvious tool wasn't the right one.

Streaming engine

DuckDB instead of Spark

The original design used Spark Structured Streaming. On a t3.small, Spark ran out of memory — the JVM overhead and shuffle machinery simply don't fit a 2 GB instance. Rather than scale the instance up (and the bill with it), I moved feature computation to DuckDB, which runs in-process with a tiny footprint. The tradeoff is explicit: I gave up horizontal scalability I wasn't using for a single-node engine that fits the hardware. For this data volume, distributed compute was overhead, not capability.

Ingestion

Kafka as a buffer, not as scale theater

Order-book updates are bursty. Kafka sits between the WebSocket consumer and feature computation so a burst doesn't back-pressure or drop data — the consumer writes at line rate and the processor drains at its own pace. It's a single-broker setup; the point is decoupling and durability, not a giant cluster.

Model

LightGBM over a deep net

The features are tabular microstructure signals — order-book imbalance, spread, depth. Gradient-boosted trees are the right fit: fast to train, fast to score, strong on tabular data, and cheap enough to retrain on the instance. A neural net would add latency and training cost for no clear gain here.

Operations

Observability and drift detection built in from the start

A streaming model that silently degrades is worse than no model. Prometheus + Grafana expose system and model metrics, and a drift check compares live feature distributions against training and fires an AWS SNS alert when they diverge. This is the difference between a demo and something you'd trust running unattended.

Delivery

GitHub Actions for CI/CD

Build, test, and the retrain/deploy path are automated through GitHub Actions so a change goes from commit to running on EC2 without manual steps — and so the pipeline is reproducible rather than a hand-configured box.

05

What broke, and what I'd do differently

The parts that didn't go to plan are the parts I learned the most from.

  • Spark OOM on a small boxThe migration to DuckDB was driven by a real failure, not a preference. Lesson: match the tool to the hardware and the data volume, not to the résumé.
  • Crossed-book artifactsRaw order-book snapshots occasionally showed a bid above the ask — invalid states that had to be filtered before feature computation, or they'd poison the model. Data quality before modeling.
  • Config gotchasSPARK_LOCAL_IP binding and case-sensitive Spark SQL cost real debugging time — the unglamorous reality of running streaming infra.
  • EC2 disk and environment hygieneRan into disk pressure and venv/Anaconda conflicts on the instance; learned to keep the runtime environment clean and provisioned deliberately.
