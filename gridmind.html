GridMind is a retrieval system for NERC CIP cybersecurity compliance documents. Rather than stop at retrieval or LLM metrics — as most RAG portfolios do — I ran a pre-registered A/B experiment asking whether an obligation classification signal measurably improves downstream ranking. The two-stage architecture assigns obligation labels to corpus chunks, feeds them into multiplicative priors after Reciprocal Rank Fusion, and measures the ranked output by Recall@5 and MRR. The result is small, with transparent analysis of the mismatch between hypothesis and observed mechanism: a +0.038 MRR gain driven by an indirect effect the pre-registration did not predict. Full technical writeup on GitHub, including reproducibility contracts (LogReg OOF F1=0.913, baseline rrf+priors R@5=0.750 consistent across runs under fixed seeds).

01

The problem

A compliance engineer asked to demonstrate conformance with one NERC requirement often has to open several 30–60 page standards, cross-reference them, check which revision is current, and judge whether each clause is mandatory ('shall') or advisory ('should') — today done with Ctrl-F over PDFs. Generic RAG makes it worse: it hallucinates citations, ignores document versions, and treats binding and advisory language identically. GridMind targets exactly that gap.

02

Architecture

An offline indexer builds the corpus; an event-driven online pipeline ingests it idempotently; a hybrid retriever serves it. The data path below is the indexing-and-serving spine.

Shadow-column A/B — obligation_strength_v2 populated from LogReg OOF predictions, swappable at query time without changing retrieval code
Cross-reference expansion — NetworkX citation graph pulls related standards into the candidate pool
Answer synthesis — Gemini Flash, called through an OpenAI-compatible endpoint with the provider set via env config, not hardcoded

▸ Postgres + pgvector + FTS is the development stack — the deployed Space runs an embedded numpy/BM25 backend (see the migration section below)

Ingestion is a single Python pipeline: NERC CIP PDFs are parsed, split into requirement-aligned chunks, tagged with a regex-derived obligation label, embedded with bge-small-en-v1.5, and upserted into Postgres. At query time the development retriever runs dense (pgvector HNSW) and sparse (Postgres full-text) searches independently, fuses their ranked lists using Reciprocal Rank Fusion, then applies freshness and obligation-strength priors as multiplicative post-fusion adjustments. A shadow column lets classifier-driven priors be swapped in for the regex-derived defaults without touching retrieval code — this is the substrate that made the pre-registered A/B experiment possible. The synthesis step calls Gemini Flash through an OpenAI-compatible endpoint, with the provider chosen by environment variable rather than hardcoded. The configuration above is the development setup; the deployed instance on Hugging Face Spaces replaces the Postgres-backed retriever with an embedded, in-process equivalent — see the migration writeup below.

03

Key decisions & tradeoffs

Four decisions carry the retrieval-impact story. Each has a defensible alternative — the choice matters because it constrained what could be measured.

  • Reciprocal Rank Fusion over weighted linear combinationRRF fuses dense and sparse rankings by rank position, not raw score, so the two retrievers do not need calibrated score scales. A weighted linear combination (60% semantic + 25% keyword + …) is simpler on paper but requires tuning weights that drift as the corpus grows. RRF is parameter-light and empirically competitive on small corpora.
  • Multiplicative post-fusion priors, not additive or in-scoreFreshness and obligation priors multiply the RRF score after fusion, rather than being folded into the retrieval scores themselves. This keeps priors ablatable — the retrieval evaluation can turn priors on and off without retraining or re-embedding — and it is the substrate that made the classifier-driven A/B experiment possible without changing retrieval code.
  • Out-of-fold predictions for the retrieval-impact experiment, full-fit for productionThe retrieval A/B uses OOF predictions from 5-fold cross-validation to avoid label leakage: each chunk's predicted label was generated by a model that never saw it. This is the academically correct choice for measuring the causal effect of classifier signal on ranking. A production deployment would use a full-fit model on all 96 labels; the choice is explicitly noted in the writeup so nothing is hidden.
  • Binary obligation classifier, not four-wayThe classifier predicts obligation vs. non-obligation, mapped back to the four-value retrieval prior via a rule-based mapping (regex modality kept where the classifier agrees; classifier-positive chunks with no regex modal default to should as a retrieval weight, not a semantic claim). Training a four-way classifier would require re-labelling and a much larger dataset. The binary choice is honest about what supervision the labels can support.
04

Serving migration: Postgres → embedded

The development stack above was built against Postgres + pgvector + FTS — HNSW for dense search, GIN/tsvector for sparse, fused by RRF. Deploying it hit a constraint the dev environment never had to face: zero-budget hosting on a single-container platform, with no managed Postgres available to it. Rather than pay for a database to serve a 96-chunk corpus, the serving path was re-architected to run in-process — numpy cosine similarity over exported embeddings in place of pgvector, and BM25 (rank_bm25) over the same chunks in place of Postgres full-text search. The fusion logic — Reciprocal Rank Fusion, then the same multiplicative obligation and freshness priors — was ported unchanged.

The migration was gated on the existing 10-question benchmark, re-run against the embedded backend and tagged benchmark-v2: Recall@5 moved 0.750 → 0.800, MRR moved 0.583 → 0.637, measured against the classifier-priors configuration (rrf+priors_v2) that was already the production baseline.

Read this as "no regression, plausibly better," not as a proven improvement. At n=10 questions, a swing of this size is within what a single question's re-ranking can produce — it is not statistically distinguishable from noise. The result was a trade, not a clean win: q004, q006, and q009 improved, while q008 regressed, its gold chunk falling out of the top 5 under the embedded backend. The benchmark supports "the migration didn't break retrieval, and plausibly helped." It does not support "the embedded backend is better."

05

Results

Two evaluation stages, each with its own protocol and reproducibility contract.

  • Obligation classification: LogReg F1 = 0.9135-fold stratified out-of-fold cross-validation on 96 hand-labelled chunks (23 positives). Compared under an identical protocol against regex (F1 = 0.727), DistilBERT with default loss (F1 = 0.516, mode-collapsed on 76-example folds with 4:1 imbalance — documented as an ablation), and DistilBERT with class-weighted loss (F1 = 0.792, recovering most of the LogReg gap by changing only the loss function). LogReg wins on aggregate F1 and is used as the production obligation classifier.
  • Retrieval baseline: Recall@5 = 0.750, MRR = 0.54510 questions with multi-chunk gold sets, evaluated across five retrieval strategies evaluated as progressively richer configurations (lexical, dense, RRF, RRF + priors, RRF + priors + cross-reference expansion). RRF + priors is the selected configuration for evaluation and downstream experiments. Baseline reproduces consistently across runs under fixed seeds.
  • Classifier-driven retrieval priors: ΔMRR = +0.038, ΔRecall@5 = 0.000Pre-registered A/B experiment: substitute LogReg OOF predictions for the regex-derived obligation prior, run the same 10-question eval, compare to the baseline. MRR improved; recall did not. Of 33 chunks whose priors changed, only 2 were in a gold set — meaning the recall boundary was never crossed. The +0.038 MRR came primarily from an indirect effect (see section 06).
  • Reproducibility contractsThe LogReg OOF export script asserts F1 = 0.913 before writing predictions; if the assertion fails, no output is produced. The retrieval baseline reproduces Recall@5 = 0.750 and MRR = 0.545 consistently across runs under fixed seeds — verified after the v2 plumbing change, which was neutral for existing strategies by design.
06

What broke, and what I'd do differently

Six findings worth naming: three from the original evaluation — the experiment mechanism, model choice on small data, and the limits of representation — and three surfaced by the Postgres → embedded migration.

  • The pre-registered mechanism did not occur; a different one didThe hypothesis was that classifier-driven priors would rescue a known failure — a question where the correct chunks describe exemption scope in asset-class language and the retriever ranks them in the 20s. That rescue did not happen; a 1.15× multiplicative boost is not enough to move a chunk twenty positions. The measured +0.038 MRR came from a different route: two other questions gained rank because non-gold distractor chunks around them were demoted, letting gold chunks climb by default — consistent with indirect distractor demotion rather than direct target promotion. Naming this openly is what makes the +0.038 legible — small, real, and mechanistically explained rather than headlined.
  • DistilBERT with default cross-entropy collapsed to majority-class predictionTwo of five folds hit F1 = 0.000 on 76-example training splits with a 4:1 class imbalance. Documented as an ablation rather than discarded; the class-weighted follow-up (F1 = 0.792, only the loss changed) is the direct comparison that isolates what supervision weighting actually buys on this data scale. Lesson: this is not an apples-to-apples comparison until the loss functions are aligned.
  • Three model families fail on the same two chunksbge-small in retrieval, LogReg in classification, and DistilBERT v2 in classification all misclassify or fail to rank the same two CIP-013 exemption chunks. Both express exemption scope as asset-class descriptions ("Cyber Assets associated with communication networks and data communication links") rather than as requirement language. The consistent failure suggests the fix is representational — query rewriting, embedding fine-tuning for exemption patterns, or cross-encoder reranking — not further prior tuning or classifier retraining on the same features.
  • The exemption blind spot is representation-specific, not corpus-wideThe same CIP-013 exemption chunks that bge, LogReg, and DistilBERT all fail on partially surfaced once BM25 entered the fusion mix in the embedded backend — question q007's MRR moved 0.0 → 0.125. BM25 doesn't share the failure mode of the learned representations, which is evidence the blind spot lives in embedding/classifier feature space specifically, not in the text itself.
  • Citation faithfulness tracks retrieval depth, not just retrieval qualityAt k=3 the LLM produced an answer with no citation at all — the chunk it needed had ranked just outside the context window. At k=5, the same question returned the citation correctly. Retrieval itself didn't change; only the serving parameter did. This is what motivates a citation-precision eval on the roadmap, rather than treating groundedness as solved once retrieval numbers look good.
  • The relevance gate needed recalibrating after the migrationEmbedded-backend RRF scores run on a different scale than the Postgres-era scores the gate was originally tuned against, so the real-vs-garbage-query threshold moved from 0.015 to 0.021. Recalibration was done on 8 real/garbage query pairs with only a 0.002 separation gap between them — a narrow margin, noted here rather than presented as settled.
07

What's next

Three items on the roadmap, in order.

  • Expand the evaluation benchmark to 30–50 questionsFrozen corpus, retrieval configuration, and classifier — the expansion tests whether the ranking results are robust to sampling variance, not whether a new system version performs better. The 10-question benchmark is tagged benchmark-v1 (Postgres era) and benchmark-v2 (same 10 questions, re-run to gate the embedded-backend migration); expansion adds new questions on top of both rather than replacing them.
  • Investigate representational fixes for the exemption-language blind spotThe cross-model failure identified above is unlikely to be fixed by further prior tuning or classifier retraining on the same features — and BM25's partial recovery on q007 (section 06) narrows the direction further, toward query rewriting or reranking rather than the embedding/classifier features themselves. Candidate directions: query rewriting to reference exemption scope explicitly, embedding fine-tuning on regulatory exemption patterns, or a cross-encoder reranker that attends to the full query-chunk pair.
  • Add a citation-precision eval, not just retrieval Recall/MRRThe answer-synthesis layer (Gemini Flash, live in the deployed demo) already produces cited answers, but section 06 showed citation faithfulness can fail even when retrieval numbers look fine — the same question came back uncited at k=3 and cited at k=5. The next eval axis is whether cited chunks actually support the claim being made, not just whether they were retrieved.
